# IshaCheck 系統完整介紹與使用前導覽

本文件提供給準備導入、測試或使用 IshaCheck 系統的人員閱讀。內容會從系統用途、適用情境、主要功能、角色權限、部署需求、首次啟用、日常操作、安全維護到備份觀念做完整說明，協助使用單位在正式上線前建立共同理解。

## 1. 系統定位

IshaCheck 是一套以「設備巡檢、場地租借、異常追蹤、報表留存」為核心的內部管理系統。系統以 Web 介面提供操作，使用者可透過電腦、平板或 kiosk 裝置進行登入、填報、查詢與管理。

本公開版本為後端可運行版本，適合用於：

- GitHub 分享與程式碼展示
- 本機測試與功能驗證
- 導入前評估
- 內部二次開發
- 建立正式部署包前的基礎版本

系統預設以 Python Web 服務啟動，資料儲存在 SQLite 資料庫中。首次啟動時會自動建立必要資料表、預設設備、預設角色權限與初始管理帳號。

## 2. 適用情境

此系統適合下列管理場景：

- 訓練場地或作業場域需要每日巡檢紀錄
- 設備需要月檢、異常紀錄、修復追蹤
- 管理者需要匯出 PDF 或 Excel 報表留存
- 場地需要提供租借申請、審核、取消與查詢
- 單位需要以角色權限區分管理員、主管、檢查人員與租借使用者
- 現場需要 touch/kiosk 形式的快速填報入口

## 3. 主要功能總覽

### 3.1 帳號登入與權限控管

系統提供帳號登入、角色分流與功能權限控管。不同角色看到的功能會依權限而不同。

預設角色包含：

- 管理員：可管理所有功能、帳號、權限、設備、報表、租借與系統設定。
- 主管：可查詢巡檢資料、匯出報表、處理異常、審核租借。
- 檢查登錄人員：主要負責每日與每月檢查填報、查詢自身或授權範圍紀錄。
- 場地租借人員：主要負責查看可租借時段、提出租借申請、查看申請狀態與提出撤銷。

### 3.2 設備巡檢

系統支援每日檢查與每月檢查。管理者可建立設備、設定設備類型、啟用每日或每月檢查，並設定每台設備的檢查項目。

巡檢填報通常包含：

- 設備代碼與設備名稱
- 檢查日期
- 檢查人員
- 各項檢查結果
- 備註或改善內容
- 異常註記
- 簽名資料

### 3.3 異常追蹤

當檢查結果出現異常時，系統可集中呈現在異常查詢中心，方便主管或管理員追蹤尚未修復的項目。

異常管理重點：

- 可依每日或每月紀錄查詢
- 可依設備、日期、修復狀態篩選
- 可標記異常已修復
- 管理者登入後可看到未處理異常提醒

### 3.4 報表與紀錄

系統提供巡檢紀錄查詢與報表匯出功能，方便留存紙本或電子檔。

常見報表包含：

- 單日單設備每日檢查報表
- 月份每日檢查彙整報表
- 單月月檢報表
- 年度月檢報表
- 年度每日檢查報表
- Excel 完整紀錄匯出

報表匯出檔案屬於執行期資料，正式使用時應納入備份，但不應提交到 GitHub。

### 3.5 場地租借

租借模組提供場地可用性查詢、申請、審核、撤銷與報表功能。

主要流程：

1. 租借人員查看場地可用性。
2. 租借人員送出申請。
3. 主管或管理員審核申請。
4. 系統保留租借紀錄與審核紀錄。
5. 如需取消，租借人員可提出撤銷申請。
6. 管理端可審核撤銷與匯出租借報表。

租借模組也包含場地管理、阻擋日期、衝突場地例外開放、租借帳號管理等進階功能。

### 3.6 Email 與提醒

系統支援 Email 設定與提醒流程，可用於檢查提醒、租借提醒或報表寄送。正式部署時需由管理者設定 SMTP 主機、帳號、密碼、寄件人與收件人資訊。

Email 密碼與第三方服務金鑰都屬於機敏資料，應只放在部署主機本地，不可提交到版本控制。

### 3.7 備份與還原

系統包含備份中心與排程備份概念，可備份資料庫、匯出報表、log、設定檔、簽名資料與其他執行資料。

建議正式環境至少保留：

- 本機每日備份
- 外部磁碟或雲端異地備份
- 還原前安全快照
- 定期測試還原流程

### 3.8 系統監控與事件紀錄

系統可記錄登入、填報、修改、刪除、備份與管理操作等事件，提供稽核與追蹤使用。

公開分享版本已移除既有資料紀錄；正式使用後產生的事件紀錄屬於營運資料，應由使用單位自行保存與管理。

## 4. 使用者導入前需要準備

### 4.1 主機與系統需求

最小測試環境：

- Python 3.11 以上，建議 Python 3.12
- Windows 10/11 或 Linux
- 2 CPU cores
- 4 GB RAM
- 30 GB SSD

正式環境建議：

- 4 CPU cores 以上
- 8 GB RAM 以上
- 100 GB SSD 以上
- 穩定網路
- UPS 不斷電設備
- 定期備份位置

### 4.2 現場設備規劃

若要用於現場填報，建議準備：

- 一台後台管理主機
- 一台或多台現場平板/kiosk 裝置
- 固定網路或穩定 Wi-Fi
- 管理者操作電腦
- 備份磁碟或雲端備份空間

### 4.3 導入前資料盤點

正式上線前，建議先整理：

- 設備清單
- 設備代碼命名規則
- 設備類型
- 每日檢查項目
- 每月檢查項目
- 場地清單
- 租借時段規則
- 使用者帳號與角色
- 報表保存規則
- 備份保存天數

## 5. 安裝與啟動

### 5.1 建立虛擬環境

Windows:

```powershell
python -m venv .venv
.\.venv\Scripts\activate
```

Linux:

```bash
python3 -m venv .venv
source .venv/bin/activate
```

### 5.2 安裝套件

```bash
pip install -r software/requirements.txt
```

### 5.3 啟動服務

```bash
python software/start.py
```

啟動後開啟：

```text
http://127.0.0.1:5000/login
```

### 5.4 初始登入帳號

測試用預設帳號：

```text
Username: admin
Password: admin123
```

正式環境上線後，請立即變更管理員密碼。

## 6. 首次啟用建議流程

建議第一次使用時依序完成：

1. 使用 `admin/admin123` 登入。
2. 立即修改管理員密碼。
3. 建立正式使用者帳號。
4. 檢查角色權限是否符合現場分工。
5. 建立或修正設備清單。
6. 設定每日與每月檢查項目。
7. 建立場地清單與租借規則。
8. 設定 Email 通知。
9. 設定備份路徑與備份週期。
10. 執行一次測試填報、查詢、匯出與還原演練。

## 7. 日常使用流程

### 7.1 檢查人員

檢查人員通常每天進入 touch 頁面，選擇設備後填寫每日檢查。若設備需要月檢，則依月份進行月檢填報。

### 7.2 主管

主管主要查看檢查紀錄、追蹤異常、匯出報表，並處理場地租借審核。

### 7.3 管理員

管理員負責帳號、權限、設備、檢查項目、備份、Email、租借設定與系統狀態維護。

### 7.4 租借人員

租借人員可查看可租借日期與時段，提出租借申請，並追蹤自己的申請狀態。如需取消，可提出撤銷申請。

## 8. 資料與安全注意事項

### 8.1 不應提交到 GitHub 的資料

以下資料屬於執行期或機敏資料，不應放入公開版本：

- `database.db`
- `software/logs/`
- `software/exports/`
- `software/secrets/`
- `software/static/signatures/`
- `.app_secret`
- `.bootstrap_credentials.json`
- 第三方服務金鑰
- SMTP 密碼
- 真實使用者資料
- 真實巡檢紀錄
- 真實租借紀錄

### 8.2 密碼與登入保護

系統有登入失敗次數限制。若短時間內多次輸入錯誤，會暫時鎖定登入，避免暴力嘗試。

管理建議：

- 正式環境不要使用預設密碼
- 每位使用者使用個人帳號
- 離職或不再使用時停用帳號
- 管理員帳號應限制人數
- 定期檢查事件紀錄

### 8.3 憑證與金鑰

Google Calendar、SMTP 或其他第三方服務憑證應由部署端自行配置，並放在本機安全路徑。若憑證曾暴露，請立即重新產生或輪替。

## 9. 測試與驗證

系統提供最小登入 smoke test，可驗證服務是否能啟動與登入。

```bash
python software/scripts/smoke_login.py
```

測試成功時會看到：

```text
PASS: login smoke test succeeded with admin/admin123
```

若測試失敗，請先確認：

- 服務是否已啟動
- `127.0.0.1:5000/login` 是否可開啟
- `admin` 密碼是否已被變更
- 是否因登入失敗次數過多被暫時鎖定
- 是否使用了不同的資料庫路徑

## 10. 正式上線前檢查表

上線前建議確認：

- 已變更預設管理員密碼
- 已建立正式帳號與角色
- 已確認設備清單與檢查項目
- 已完成租借場地與規則設定
- 已設定備份路徑
- 已測試備份與還原
- 已設定 Email 通知
- 已確認公開網址或內網網址
- 已測試登入、填報、查詢、匯出、租借申請與審核
- 已確認機敏資料未提交到 GitHub

## 11. 常見問題

### Q1：第一次啟動後找不到資料庫？

系統首次啟動會自動建立 `database.db`。如果沒有看到資料庫，請確認服務是否成功啟動，以及目前工作目錄是否正確。

### Q2：`admin/admin123` 無法登入？

可能原因包含密碼已被修改、資料庫不是目前這份、登入失敗次數過多被暫時鎖定，或服務尚未重啟。可先執行 smoke test 協助判斷。

### Q3：可以直接把整個資料夾上傳 GitHub 嗎？

不建議直接上傳執行後的資料夾。請先確認 `database.db`、`logs`、`exports`、`secrets`、簽名圖與憑證都沒有被加入版本控制。

### Q4：正式使用時資料要放哪裡？

可使用預設資料夾，也可透過環境變數調整資料、設定、log、匯出、secrets 的位置。正式部署建議將資料與程式碼分開管理。

### Q5：這套系統能多人使用嗎？

可以，多位使用者可透過瀏覽器登入使用。正式環境需注意主機效能、網路穩定性與備份策略。

## 12. 建議導入方式

建議採用分階段導入：

1. 本機測試：確認能安裝、啟動、登入。
2. 管理者試用：建立設備、帳號、檢查項目與租借設定。
3. 小範圍試行：讓少數人員填報與審核。
4. 資料檢核：確認報表、異常、備份符合需求。
5. 正式上線：開放所有使用者使用。
6. 定期維護：檢查備份、帳號、事件紀錄與系統狀態。

## 13. 結語

IshaCheck 的設計重點是讓現場檢查、設備管理、異常追蹤與場地租借流程集中在同一套系統中。對準備導入的單位而言，最重要的是先整理好設備、角色、流程與備份策略，再逐步把日常作業搬進系統中。

公開版本適合展示與測試；正式使用時，請務必妥善管理密碼、資料庫、log、匯出報表、簽名資料與第三方憑證。
